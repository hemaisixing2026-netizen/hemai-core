# Cron审计三级分类 · 2026-06-13

> 背景：合脉生态40条cron，某周六早上老刘说"排查一下是不是有停滞的"
> 关键洞察：不是所有 last_run > 12h 的 cron 都是故障。错误分类会触发假警报，浪费注意力。

## 三级分类

| 级别 | 信号 | 含义 | 动作 |
|:--|:--|:--|:--|
| 🟢 周末合理休眠 | last_run > 12h + schedule 含 `1-5` | 工作日cron，周六日不触发 | **不动**。不是故障。 |
| 🟡 定时未到 | last_run < 24h + next_run 在未来 | cron正常，只是还没到下一次触发 | **不动**。等触发窗口。 |
| 🔴 真正停滞 | last_run > 24h + schedule 应已触发 + 非weekday-gated | cron错了或产出异常 | **诊断+修复** |

## 6/13实战

```
引擎1·业务全扫: last_run 13h前 → 但schedule是 0 10,14,16,20 * * 1-5
→ 周六不触发 → 🟢 合理休眠，不动

蜕骨·野心巡航: last_run 00:18 error → 下次12:00
→ 上次报错但定时在 → 🟡 等12:00自动重跑

云途追踪: last_run 08:04 ok → 包裹6/11已签收
→ cron在空跑 → cronjob pause
```

## 关键原则

**看到数据文件 >20h stale → 第一步不是诊断故障，是查cron schedule。**
- 这个文件有没有对应的cron？
- cron的schedule是什么？
- 下次触发什么时候？
- 上次触发正常吗？

只有确认cron已触发但未产出/报错，才算真正的故障。否则是"cron还没到触发窗口"——这是正常状态，不是停滞。
